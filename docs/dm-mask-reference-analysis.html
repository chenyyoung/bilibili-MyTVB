新增结论（本次深挖后的重大修正）：
  1. 参考的 mask 真正渲染消费者是 Chronos 跨进程，主 APK 自身并不下载/解析/渲染 webmask。RemoteServiceHandler 只是把 DmViewReply 通过 GetDmView RPC 转发给 Chronos 端（独立 native 进程 libchronos.so），并在 Chronos 协议响应里写入 setMaskEnabled() / setBlockMask()。
  2. 关键证据：lib/arm64-v8a/libchronos.so 是独立 native 库；lib/androidx.graphics.path.so 提供 path 几何加速；全 smali 中 DmViewReply.hasMask() 仅 1 次调用（Bi1/a.smali，仅用于 dm_mask 上报），VideoMask.getMaskUrl() 0 次调用。

  4. “渲染查询用 player clock，但预加载用 video frame PTS”的双时间源问题仍是真实风险：MyBLBL 已在 DmMaskController.onVideoFrameAnchor() 留空实现，VideoFrameMetadataListener 已不再注册，但该入口未彻底删除，建议在下一轮重构中直接删掉避免未来误用（原文档“保留为空兼容入口”是带风险的妥协）。

6. webmask 文件语义

我们当前解析的 webmask 结构和参考数据字段是吻合的：

  1. 头部：MASK 魔数，version，reserved，segmentCount。我们在 WebmaskParser.parseSegmentCount() 解析。
  2. 索引表：每段 timeMs + byteOffset，每条 16 字节。我们在 parseSegmentMeta() 解析为 LazyMaskSegment。
  3. 段数据：每段 gzip，解压后是二进制帧流：uint32 payloadLength + uint64 presentationTimeMs + payload；payload 才以 data:image/svg+xml;base64, 开头。
  4. 帧内容：每帧 SVG path 表示允许弹幕绘制的区域，人物/主体区域是洞，需要 EVEN_ODD 填充规则。注意边界（本次深挖后补充）：webmask 中可能存在 <path transform="scale(2) translate(10,20)"> 等 transform 属性，也可能 viewBox 不是 (0,0,width,height)（如 PUGV 课程视频主体在右下角），目前 WebmaskParser 只读 width/height 和 d=，未解析 transform 和 viewBox 偏移——遇到 PUGV / 非等比显示场景会出现主体错位。

7. 参考逻辑从头到尾

  1. 请求弹幕 view：播放器 resolve 阶段请求 DmView。
  2. 返回 mask 元信息：DmViewReply.mask 中携带 VideoMask。
  3. 写入互动参数：InteractLayerService.setDmViewReply() 把完整 DmViewReply 保存到 DanmakuParams。
  4. 交给 Chronos：GetDmView 响应只放 work_id/video_id，完整 protobuf 作为附件传入 Chronos。
  5. 判断能力可用：DmViewReply.hasMask() 表示当前视频有防遮挡资源。
  6. 读取用户开关：设置 key 为 DanmakuMask，默认 true。
  7. 加载 mask 文件：用 maskUrl 下载 webmask，按 fps 建立时间线。
  8. 跟随播放器 clock：RemoteServiceHandler 注册 addPlayerClockChangedObserver，把 current_time 和 playback_rate 发给 Chronos。
  9. 按 PTS 查帧：用当前播放器 position 定位 segment 和帧。
  10. 渲染裁剪：把 mask path 映射到视频实际显示 bounds，对弹幕层进行裁剪，人物洞内不绘制弹幕。
  11. 无遮罩语义：真实普通 video mask 中“无遮罩”表现为全矩形 <path>，不是空 SVG；解码失败不能伪造成空 paths。
  12. 缺帧语义：当前段未加载时不应拿旧 PTS 的遮罩硬顶，否则视觉上就是遮罩追不上。

9. 除了空帧填充，还有哪些差异

9.1 半帧补偿

这是明确差异。它会让 mask 查询 PTS 比播放器 position 大半个 mask frame。24fps 时约 20ms，30fps 时约 16ms。你看到“要么提前”的一部分来自这里。

9.2 round-to-nearest 选帧

round-to-nearest 的本质是在当前帧播放到一半时提前切到下一帧。mask 路径是主体轮廓，主体移动时提前切下一帧会非常明显。参考没有证据支持这种提前切帧。

9.3 缺帧复用旧遮罩

这看起来能“防闪”，但时间语义是错的。当前 PTS 查不到帧时复用旧帧，会让遮罩停在旧位置。主体继续移动，视觉上就是遮罩追不上。

9.4 当前段预解析不够及时

如果当前段还没下载/解析完成，严格不复用旧帧后会短暂无 mask。这个比旧遮罩错位更接近参考语义，但如果频繁出现，就说明预加载窗口或触发时机还需要加强。最新深挖后已确认：onVideoFrameAnchor() 不应作为预加载时间源；预加载必须跟绘制查询一样使用播放器 clock。否则段加载和画面查询会在段边界处分叉，表现为遮罩闪烁或短暂丢失。

9.5 第二时间源：video frame PTS

这是本次继续深挖后新增确认的关键差异。参考的 RemoteServiceHandler 注册 addPlayerClockChangedObserver，再发送 OnPlaybackStatusChanged(current_time, playback_rate)。我们之前虽然绘制查询已改成 player.currentPosition，但预加载仍由 VideoFrameMetadataListener.presentationTimeUs 触发。两套 PTS 在 seek、缓冲、解码排队、段边界附近不保证一致，正好解释“遮罩追不上、被挡住的弹幕闪烁、遮罩偶尔丢失”。

9.6 position interceptor 缺失

参考 clock 分发前可经过 InterfaceC37058h 修正 position。我们没有发现 Media3 侧的等价层。如果普通直播视频没有复杂时间轴，这通常不影响；但遇到片头跳过、广告、互动视频、剪辑时间轴，可能造成差异。

9.7 渲染架构不同（2026-06-06 修正）

参考的主 APK 根本不在自己进程内渲染 mask。mask 真正渲染在 Chronos 跨进程 native 端，主 APK 只发 RPC 把 DmViewReply 和播放状态送过去。最新 native 证据不是简单 clipPath，而是 libchronos.so 里的 fragment shader：读取 cron_mask_texture 的 alpha 得到 mask_sample，再执行 cron_output = CRONDefaultShading() * mask_sample。因此 MyBLBL 不能继续用父级 clipPath 当作完全等价方案；当前改为 saveLayer + DST_IN，让弹幕内容先绘制完成，再乘当前 mask alpha。

  1. ViewModel 状态：VideoPlayerViewModel.DmMaskState.Ready(maskUrl, cid, fps) 表示拿到 mask 元信息。
  2. 播放器视图接入：MyPlayerView 给 DmMaskController 注入 player.currentPosition。
  3. 加载：DmMaskController.loadMask() 调用 DmMaskRepository.downloadAndParse()。

16. 时序层盲区：floor + 不预取未来帧 = 延后 1 帧

原文档第 9 节“除了空帧填充，还有哪些差异”分析了半帧补偿、round-to-nearest 等选帧策略，没分析 floor 选帧的物理含义。这是更深的一层。

16.1 视频路径的物理时序

视频帧走独立 surface，不依赖 view 系统：

  1. t = 0 ms：decoder 解码视频帧 N，releaseNs 写入 surface。
  2. t = 0~16.67 ms：SurfaceFlinger 在 t = 16.67 ms vsync 合成帧 N 上屏。
  3. t = 16.67 ms：视频帧 N+1 释放，t = 33.33 ms 上屏。

结论：视频帧 N 从 releaseNs 到上屏延迟 1 vsync ≈ 16.67ms（60Hz）。

16.2 mask 路径的物理时序

mask 走 View 系统，DanmakuMaskHostLayout.dispatchDraw() 拿当前 player.currentPosition 查帧并 clipPath：

  1. t = 0 ms：父容器 invalidate，主线程在 t = 0~16.67 ms 之间某时刻执行 dispatchDraw。
  2. 读 player.currentPosition，floor 查 mask frame。
  3. saveLayer + super.dispatchDraw + DST_IN 提交给 RenderNode。
  4. t = 16.67 ms vsync：mask + 弹幕上屏。

16.3 时序对照表

时刻 (vsync)视频上屏mask 上屏（floor + 不预取未来帧）视觉一致性
0（准备 release）（尚未合成）—

2 (33.33ms)视频帧 N+1mask(frame_for_position_N)（因为 floor 还在取 frame N）❌ mask 落后 1 帧 ≈ 16.67ms

“1 vsync 物理对称性自动消解”这一说法（mask.md 原文）只在 vsync=1 成立，之后每一帧 mask 都延后 1 帧。

16.4 三种选帧策略对比

原文档第 9.2 节说“round-to-nearest 会在两帧中点后提前切下一帧”，分析对；但没分析 floor 也有 50% 时间延后的问题。floor 是“参考行为”，但不是“完美对齐”。

16.5 真正消除 1 帧延后需要做的

如果想 mask 跟视频像素级对齐，必须让 mask 查询的 PTS = “即将上屏的”视频帧 PTS，而不是“当前已上屏的”视频帧 PTS：

  • 主线程在 t = 0 时刻 dispatchDraw，提交后上屏是 t = 16.67 ms——mask 应该查 player.currentPosition + 16.67 ms 处的 frame。
  • 但 16.67ms 是屏的 vsync 周期，要从 Display.refreshRate 算出来，不能写死 60Hz 假设。
  • 120Hz 屏 = 8.33ms，144Hz 屏 = 6.94ms。这个数会改变 mask 查询的 PTS 偏移。

原文档“通过同一 clock 的预加载减少缺帧”是更现实的解法——不去解决 1 帧延后（因为没有 100% 解），而是减少跨段空窗（人眼对“突然没遮罩”比“延后 1 帧”敏感）。

19. webmask 解析边界 bug

原文档第 6 节描述了 webmask 文件结构，没分析解析时的边界 bug。下面是 WebmaskParser 当前实现里的潜在问题。

19.1 段内帧 PTS 的整数除法漂移

val ptsMs = if (totalFrames > 1 && segDurationMs > 0L) {
    segment.timeMs + (localIdx.toLong() * segDurationMs) / totalFrames
} else if (fps > 0) {
    segment.timeMs + localIdx.toLong() * 1000L / fps
} else {
    segment.timeMs
}

实际例子：30fps 段长度 6500ms，totalFrames=195，localIdx * 6500 / 195：

  • localIdx=0 → 0ms
  • localIdx=1 → 33ms（理想 33.33）
  • localIdx=2 → 66ms（理想 66.67）
  • …
  • localIdx=194 → 6466ms（理想 6466.67）

单帧差 0.33ms，195 帧累积差 ≈ 0ms（数学上会抵消）——实际影响不大，但当段内 fps 跟 mask fps 不一致时（罕见但发生过），会系统性偏移。

建议：改为浮点除法 + 取整，或读 webmask 段的 frameDuration 字段（如果存在）。

19.2 H / V 命令的状态机风险

svgPathToAndroidPath 的 H 分支：

'H' -> {
    currentX = absoluteX(tokens[i])
    path.lineTo(currentX, currentY)
    resetSmoothControls()
    i += 1
}

SVG path d 字符串里 H 100 200 300 是三个值共用一个 H 命令。当前实现读完一个值后 i += 1，下一个数字 token 进入 default 分支，触发 i++，回到 while 头部，currentCommand 还是 H——能 work，但完全靠"default 分支 return false 后回到外层 while 重新 dispatch"，可读性极差且容易出 bug。

建议：把 H/V/L/M 改成“内层 while 消费所有数字 token”。这是 30 行代码的改动，但能避免未来在 c/q/t 改命令时漏改。

19.3 SVG transform 属性未解析

webmask 里可能存在 <path transform="scale(2) translate(10,20)">。当前 WebmaskParser 只读 d=，不解析 transform。这意味着：

  • 横竖屏切换：人物 path 在 SVG 坐标系里可能被整体缩放/平移，我们没意识到这点。
  • 非等比显示：surface bounds 跟 SVG viewBox 不一致时，transform 差异会被放大。

建议：用 Android androidx.core.graphics.PathParser 解析 transform 字符串，或自己写一个 Matrix 应用层。

19.4 MaskFrame 缺 viewBox 偏移

当前 MaskFrame(presentationTimeMs, paths, svgWidth, svgHeight) 假定 SVG viewBox 起点在 (0,0)。但 PUGV 课程视频可能用 viewBox="100 50 320 180"——人物区域在 (100,50) 起点，当前实现会按 (0,0) 起点映射到视频 bounds，主体错位 100 像素左右。

建议：扩展 MaskFrame 增加 viewBoxX、viewBoxY 字段，DanmakuMaskHostLayout 渲染时把 dx = bounds.left - viewBoxX * sx 算回去。

19.5 BOM 头与 0 byte body

当前解析流程不防 BOM：

  • HTTP 响应如果带 UTF-8 BOM（极罕见但发生过），Base64.decode 会把 BOM 当 base64 数据，SVG 解析失败。
  • CDN 200 OK + 0 byte body 时 parseSegmentCount 返回 -1，但 parseSegmentMeta 之前已经走通头部解析，行为不一致。

建议：

if (bytes.isEmpty()) {
    AppLog.e(TAG, "Empty body for range $start-$end")
    return null
}
// SVG 解析前剥 BOM
val svgText = if (svgText.startsWith("\uFEFF")) svgText.substring(1) else svgText

19.6 gzip 流式解压 OOM 风险

parseSegmentFrames 用 ByteArrayOutputStream 一次性读完 gzip 输出。普通段 ≈ 100KB 没问题，但极端长视频的某段可能 5MB+，一次性内存分配会触发 OOM。

建议：改用 GZIPInputStream 直接喂给 Base64.decode 包装的 InputStream，边读边解析，避免一次性 byte[] 分配。

20. 状态机边界

原文档第 10 节描述了 MyBLBL 链路，没分析状态转换时的边界问题。下面 6 个状态机边界在用户场景里高频触发，但当前实现都没特殊处理。

20.1 ExoPlayer STATE_BUFFERING

播放器进入 BUFFERING 时：

  • player.currentPosition 不变（停在最后一个能播到的位置）。
  • 主线程 dispatchDraw 还在跑（如果 host 还可见）。
  • 视频 surface 不再 release 新帧（停在旧帧）。

当前 DmMaskController.currentVideoPtsMs() 总是返回 player.currentPosition → mask 钉在旧 PTS。但视频也钉在旧帧，所以对得上。

风险：BUFFERING 期间 host 还在重绘（虽然内容不变），会浪费 CPU。建议：在 setPlaying(false) 或 STATE_BUFFERING 回调里停止 Choreographer 循环。

20.2 playbackSpeed 切换

fun setPlaybackSpeed(speed: Float) {
    if (speed.isFinite() && speed > 0f) {
        onPlayerClockChanged(speed, currentVideoPtsMs())
    }
}

速度切换时：

  • clockPositionMs = currentVideoPtsMs()，重置 clockRealtimeMs——外推窗口清零，对的。
  • 但 lastPreloadedSegIndex = -1 没 reset——2.0x 播放跨段速度变快时，maybePreloadAroundCurrentPts 不会重新触发预解析。

建议：在 setPlaybackSpeed 里 lastPreloadedSegIndex = -1。

20.3 Seek 后的 0~200ms 窗口

onSeek() 流程：

  1. awaitingSeekReady = true，lastPreloadedSegIndex = -1。
  2. clearMask() → invalidate。
  3. 等 playbackReady + seekReadyAt → 重新开始。

2026-06-06 修正：这段早期建议已废弃。旧建议“seek 后冻结旧帧 100ms”会把旧 PTS 遮罩套到新画面上，正是“遮罩追不上”的来源之一。当前实现改为 onSeek() 立即 clearCachedMask()，SEEKING 期间 shouldRenderMask=false，等播放器 ready 后再按当前 player.currentPosition 查当前帧。

20.4 横竖屏切换

转屏动画期间 dispatchDraw 拿旧 videoBounds：

  • videoBoundsProvider 返回 host view 的实时 bounds，转屏动画期间这个 bounds 跟视频 surface bounds 不一致。
  • mask path 按旧 bounds 缩放，绘制到 host view 上 → 视觉上“mask 不跟视频走”。

当前 DanmakuMaskHostLayout 在转屏期间不主动 invalidate，所以“mask 钉在旧位置”直到下次 invalidate。

建议：在 onConfigurationChanged 里强制 invalidate 一次。

20.5 后台回前台 / UI 卡顿

旧实现曾用 JankMonitor 统计 Choreographer 间隔，并在持续卡顿时自动关闭 mask：

  1. 后台期间 vsyncIntervalEmaNs EMA 被脏数据污染（间隔 1 秒+）。
  2. 回到前台后 vsyncIntervalEmaNs 还是大值。
  3. 5 秒后 triggerAutoDisable() 把 mask 关掉。

这是参考没有的行为，会制造整块人物区域突然露出的假象。最新实现已移除自动关闭逻辑；Choreographer 只负责 ACTIVE 状态下持续 invalidate mask host。

20.6 弹幕全过滤导致 host 不重绘

如果用户开启了关键词屏蔽（DmMaskWall），所有弹幕都被过滤掉 → 弹幕 view 不再 invalidate → DanmakuMaskHostLayout 不重绘。

场景：用户切到“弹幕蒙版全开”，mask 永远停在第一帧。如果第一帧是人物全开 mask，后续视频切到“人物关闭 mask”——但 host 不重绘，用户看到的还是“人物全开”。

建议：在 DanmakuFilter 变化监听里强制 invalidate 一次 host。

21. 原文档两个错误判断

21.1 “onVideoFrameAnchor() 保留为空兼容入口” 是带风险的妥协

原文档第 8 节写“MyPlayerView 不再注册 VideoFrameMetadataListener；onVideoFrameAnchor() 保留为空兼容入口”。本次深挖认为这个保留是带风险的：

23. 补充结论

本次深挖发现的三个真正盲区：

  1. 架构盲区：mask 真正渲染在 Chronos 跨进程 native 端（libchronos.so），主 APK 自身是 RPC stub，不是“主 APK + Chronos 共用 clock”。原文档主线的修复（floor / 不预取未来帧 / 删 saveLayer）方向对，但对架构差异的描述是过低估计。
  2. 时序盲区：floor + 不预取未来帧实际是“延后 1 帧 ≈ 16ms”，不是真正对齐。这条无法完全消除，原文档“通过同一 clock 预加载减少缺帧”是更现实的解法。
  3. 边界盲区：webmask 解析（H/V 状态机、SVG transform、viewBox 偏移）、状态机（BUFFERING / 切速 / seek / 转屏 / 后台回前台 / 弹幕全过滤）、三套 mask 系统（普通视频 / DmMaskWall / PUGV / 直播）的并存关系，原文档全都没分析。

原文档主线 1~13 章结论依然有效——空帧前向填充、半帧补偿、round-to-nearest、缺帧复用旧遮罩、第二时间源五个修复点没问题。本次深挖是在这五个点之上的“外围问题”，不替代主线修复。

28.2 MyBLBL 旧解析为什么会造成整块闪烁

旧实现用 splitBy(decompressed, "data:image/svg+xml;base64,")。这样切出来的第 N 帧尾部会包含第 N+1 帧的 12B 二进制帧头。例如第 0 帧 base64 末尾之后紧跟：

00 00 06 1a 00 00 00 00 00 00 00 21

这 12 字节不是 base64 内容；它是下一帧的 payloadLength=1562 和 pts=33。旧实现把它一起交给 Base64.decode()，再在失败时执行：

frames.add(MaskFrame(presentationTimeMs = ptsMs, paths = emptyList()))

这一步语义是错的：解码失败不是“当前没有人物”，而是“这帧解析失败”。一旦被伪造成空 paths，HostLayout 的逻辑就会走：

if (frame != null && frame.paths.isEmpty()) {
    cachedFrame = null
    super.dispatchDraw(canvas)
    return
}

于是同一段人物轮廓在有效帧时被 DST_IN 遮住，在误解析空帧时完全不遮罩，下一帧解析成功又遮住。这就是用户看到的“整个人物区域，弹幕被遮住了又露出来，又遮住”。
