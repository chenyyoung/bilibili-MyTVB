核心结论：渲染层（draw）非常健康，单帧 avg 0.2–0.6ms，远未触及 16.6ms 帧预算。 真正的问题是主线程阻塞有两处——appendData 的全量排序和 publishDanmaku 的去重发布， 外加分段范围状态机不统一导致的重复加载。
v2 修正：初版误判 appendData 排序在后台线程，经代码核实实际在主线程（withContext(Main.immediate)）， 因此它的优先级从 P2 上调到 P0。

2三层队列画像（理解优化的关键）

MyBLBL 的普通弹幕是「上游全量、下游窗口化」的双层结构。优化要优先收敛上游两层，不要绕过窗口化直接改 Runtime。

层级主要代码数据规模性能含义
加载层loadDanmakuSegmentPayloadrange/segment proto，数千到上万条网络 + protobuf parse + model 构建，IO 线程
时间线层danmakuTimeline已发布 rawTotal，日志最高 25580排序、签名、窗口二分——主线程开销集中在此
渲染窗口层buildPreparedWindow + akdanmaku runtime首批 144，滚动追加 96真正进引擎的量很小，是 draw 稳定的关键

Runtime 已有增量 frame、transition frame、cache reuse、load shed、measure scheduler 等自适应保护（MAX_MEASURE_PER_FRAME 等）。直接改引擎布局/缓存层容易和这些策略冲突，收益不如处理时间线层。

3实测瓶颈（按修正后优先级）

🔴 P0 · 最高优先

appendData 全量排序在主线程，随 rawTotal 线性恶化

代码证据： MyPlayerDanmakuController.kt:258-262 
prepareJob = controllerScope.launch {        // Default 启动
    previousJob?.join()
    val mergedRawData = withContext(Dispatchers.Main.immediate) {  // ← 切回主线程
        (danmakuTimeline.data + data).sortedBy { it.progress }     // ← 25580 条拼接+排序在主线程
    }
    val rawSignature = mergedRawData.fastRawSignature()            // ← 只有签名在 Default
}
根因
每次 appendData（约每 1–2 秒一次）都把 danmakuTimeline.data + data 全量拼接 + 全量排序，且包在 withContext(Main.immediate) 里。rawTotal 随视频时长/seek 次数线性增长（日志最高 25580），主线程开销随之恶化。
为何最高优先
它全程持续发生（不像 init 只起播一次），且越播越重。是当前最值得先修的主线程阻塞。
注
fastRawSignature（:1782）确实在 Default 后台——初版把"签名+排序"都归为后台是错的，只有签名在后台。
🔴 P1

init 发布阻塞主线程 45ms → 起播 jank

证据： debug.txt:139 danmaku_publish replace=true cost=45ms input=3000 → :144 ui_frame_jank interval=133ms
根因
publishDanmaku（VideoPlayerViewModel.kt:4226-4244）在主线程做 distinctRegularDanmaku() 去重 + danmakuIdentityKey() 字符串生成 + filter。45ms 是这些的总和，不只 HashSet（初版描述不完整）。
关键洞察
setData 把 3000 条存为 timeline，但首屏只 buildPreparedWindow 144 条窗口（initial=144 raw=3000）。所以"首屏发 3000 条"是为维护 timeline，不是渲染必须。
正确解法
拆「可渲染初始窗口」和「后台 timeline 补充」：先交 144 窗口保首帧，余量异步合并进 timeline。比单纯延后 setData 更能保护起播首帧。

🔵 观察项

frame layout 偶发 8ms · 遮罩链路未纳入基线

  • frame layout：update overload layout=8ms，偶发（seek/批量变更后），引擎内部轨道重分配，暂不动。
  • 遮罩/特殊弹幕：DanmakuMaskHostLayout / DmMaskRepository / SpecialDanmakuOverlayView 未纳入本基线。普通弹幕 draw 健康 ≠ 遮罩健康。若反馈"开智能防挡才卡"，单独看 mask_draw_summary（avgCost/rebuilds/pathBuild）和 special_danmaku_draw。

4非瓶颈澄清（避免过度优化）

误判点实测结论
渲染 draw 管线avg 0.2–0.6ms远低于帧预算，空间分区/批量绘制等引擎大改收益≈0
绘制缓存 cacheMiss仅 seek 后短暂出现命中率 >90%，多级缓存/LRU 改造无必要
BiliDanmakuFilter 过滤0–5ms / raw=96小批次低开销，正则预编译/结果缓存收益微小
二分查找 lowerBoundProgressO(log n)，144 条 ~8 次TreeMap 索引化属过度设计
轨道分配 findOrCreateRowrows ≈ 屏幕轨道数（个位）非万级遍历，无需索引
「空间分区渲染」「增量 frame」「多级 cache」「动态对象池」针对的都不是实测瓶颈，且多需改 akdanmaku 第三方引擎，风险/收益比差，明确不推荐。

PR 1 · 去掉 appendData 主线程全量排序 最高优先
主线程只取 timeline 快照；归并、排序、签名全移到 Default；旧/新数据均有序则改 O(n) 归并。加单测覆盖有序归并。解决 P0 + 顺带 P3。
PR 2 · 发布路径异步 + 首屏窗口化
把 distinctRegularDanmaku、danmakuIdentityKey、emitted list 计算移出主线程；先交付 144 窗口保首帧，余量异步合并进 timeline。解决 P1。
PR 3 · tail 延迟任务二次检查（最小修复）
在 loadAndPublishInitialTailRange 的 delay 后、请求前重新读 danmakuSegmentCoveredUntilMs[segment]：若 coveredUntil ≥ rangeEndMs 跳过；只覆盖前半段则把 rangeStartMs 推进到 coveredUntil。风险最低、立竿见影降 historyDropped。
PR 4 · DanmakuRangeCoverage 统一调度
单点 coveredUntil → 区间集合（每段维护 completed + in-flight 区间，请求前 subtract 取缺口）。seek/tail/segment 都调 requestMissingRanges。彻底解决 P2。涉及之前修过的分段逻辑，需充分回归。
PR 5 · Timeline segment blocks
rawTotal 万级后再做：timeline 改成分段有序块，窗口查询在块级索引定位，按需 materialize。从根上消除全量复制。风险最高，放最后。
关于 PR 3/4 的回归提示：它们触及之前刚修过的分段加载逻辑（弹幕快进修复）。建议作为独立 PR，重点回归：起播、自然播放跨段、快进到任意位置、续播。appendData 主线程排序修完后，historyDropped 的「丢弃」不再伴随主线程卡顿，但仍浪费网络，所以 PR 3/4 仍要做。
