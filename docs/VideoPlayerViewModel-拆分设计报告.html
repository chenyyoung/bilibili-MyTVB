VideoPlayerViewModel 模块化拆分设计

将 5098 行的巨型 ViewModel 按职责拆分为多个协作 Controller

阶段一 · 阶段二 心跳上报 弹幕系统 CDN/编码/画质 Fallback 2026-06-14

1拆分背景与策略

VideoPlayerViewModel.kt 已膨胀到 5098 行 / 230KB，含 ~140 个函数、151 个 private 状态字段、30 个嵌套数据类，承担了播放加载、弹幕、心跳、fallback、字幕、SponsorBlock 等十余项职责。项目已有「ViewModel 持有纯协作对象」的成熟范式（PlayerSessionCoordinator、VideoPlayerStreamResolver、VideoPlayerPlayInfoGateway 等），本设计沿用该范式。

实施策略：并行设计 + 串行实施

为何实施必须串行。三个模块落地时都要修改同一个文件——从 ViewModel 删除函数/字段、改构造函数、加桥接。三者还共享 cid/aid/position 等 116 处状态引用，并行实施必然互相覆盖。因此：

  • 设计阶段：已用 3 个子代理并行深挖三个模块，产出精确到函数/字段/接口/桥接点/风险点的设计（本文档即整合成果）。
  • 实施阶段：按「心跳 → 弹幕 → fallback」顺序逐个落地，每步编译通过、播放逻辑不破，再进下一步。

预期成果

2三模块共享约定

三份设计在协调点上完全一致，统一遵守：

协作模式
ViewModel 持有 Controller，ViewModel 实现 Context 接口、传 this（零 lambda），Controller 经接口回调读写共享状态。
共享上下文
播放身份/位置/时长/画质等（currentCid/currentAid/_currentPosition/_duration/currentPlayInfo 等）一律不迁移，留 ViewModel。
并发状态
各模块自己的 Job、generation、缓存、计数、签名集等随模块整体迁移，不可拆散。
数据类
嵌套 PlayableEpisode/PlaybackRequest 等暂不动；模块专属数据类随模块走（DmMaskState/DanmakuUpdate 提到 Controller 顶层 public）。
新文件位置
统一放 com.tutu.myblbl.feature.player 包，对齐已有协作类风格。
协程域
Controller 接收注入的 viewModelScope，不自建 scope，生命周期跟随 ViewModel。

5模块三 · CDN/编码/画质 Fallback 中-高风险

设计要点

  • 530 行函数迁入 PlaybackFallbackController，含错误分类、CDN/编码切换、playInfo 刷新、派发尝试、fallback 计划、画质降级六组。
  • 依赖注入 streamResolver/dashMediaSourceFactory/qualityPolicy/playInfoGateway + FallbackContext + scope。
  • 核心主链 rebuildPlayback/loadPlayUrl/requestPreparedPlayback/applyPreparedPlayback 全部留 ViewModel，Controller 经 FallbackContext 回调触发重建。

最大耦合难点：共享可变状态决策。currentDashSession / currentStreamFallbackPlan / fallbackRouteIndex / fallbackCdnIndex 被 fallback（读写）与加载主链 applyPreparedPlayback（异步重建）双写。

决策：四者统一留在 ViewModel，Controller 纯经 FallbackContext 接口读写（getter 实时读字段、setter 主线程同步落地）。这是唯一能避免迁移后双写竞态的方案。Controller 不持有这四个可变引用，只「提议」新值，由 ViewModel 落地。

迁移（fallback 专属）留 ViewModel（共享）
fallbackAttemptCount / playInfoRefreshRetryCount / attemptedFallbackSignatures / softwareDecoderDowngradeAttemptedCid / preferSoftwareDecoderSafePlayback / softwareDecoderSafeQualityId / lastPlaybackPositionMs / PlaybackErrorType 枚举 / MAX_FALLBACK_ATTEMPTS 等 2 常量 currentDashSession / currentStreamFallbackPlan / fallbackRouteIndex / fallbackCdnIndex / currentPlayInfo / selectedCodec / requestedQualityId / useDashPlayback / readSoftwareDecoderSafeQualityId()（持久化逻辑留 VM）

FallbackContext 关键回调

  • 读：dashSession / streamFallbackPlan / fallbackRouteIndex / fallbackCdnIndex（实时读 VM 字段）。
  • 写：onDashSessionUpdated(session) / onStreamFallbackPlanUpdated(plan, routeIndex, cdnIndex)。
  • 触发重建：suspend requestPreparedPlayback(...) / applyPreparedPlayback(...) / emitPlaybackRequest(...)。
  • UI 写：setSelectedCodec(codec) / clearError() / reportError(msg)。

同步原子性：handlePlaybackError/handlePlaybackStall 在 ExoPlayer 主线程错误回调中同步串联触发重建，Controller 必须保持同步语义，不能把整条链包进协程（否则丢失错误→重建原子性）。loadPlayUrlWithCurrentContext 内部才用 scope.launch。

切集 reset 合并：原 resetFallbackState()（896）与独立的 softwareDecoderDowngradeAttemptedCid = 0L（897）合并为 fallbackController.reset()，不可遗漏。

对 Fragment 保持透明：ViewModel 保留 fun handlePlaybackError(...) = fallbackController.handlePlaybackError(...) thin wrapper，VideoPlayerFragment.kt:283/1180 零改动。
